How should the CRM be structured?

Three approaches to organizing the system — pick the one that feels right

A

Flat & Simple Recommended

One CRM app. Each website is a "Site" in the database. The client logs in and sees all their sites in a sidebar. Click a site → manage its blogs and text fields. You (admin) add new sites via a simple admin panel.

Pros

  • Simple to build & use
  • Fast to add new sites
  • One deployment

Cons

  • Less separation between sites
B

Site-Scoped with Permissions

Same as A but with per-user, per-site access control. You can assign editors to specific sites only. The client owner sees all sites; their employees only see assigned ones.

Pros

  • Fine-grained control
  • Scalable if client grows

Cons

  • More complex to build
  • Possibly overkill now
C

Per-Site API Keys

Each site gets its own API key. The Next.js site authenticates with that key to pull content. Editors log into the CRM and manage all sites. Tight API-level separation.

Pros

  • Strong API isolation
  • Easy to revoke access per site

Cons

  • More DevOps overhead
  • Overkill for one client
